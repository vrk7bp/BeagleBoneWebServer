TimDuncanInspiration, Tim Duncan Inspired Emails

Summer 2014, [Independent Project]

Inspired by the heroic 2014 NBA Finals win by the San Antonio Spurs. As a way to celebrate the win, and tease my Miami Heat friend, I decided to make an e-mail application that would send a different picture of Tim Duncan (as well as a random quote) to anyone who chose to sign up for the e-mail list. I used the Custom Google Search API and SendGrid API to set this up, and am currently running on an AWS micro instance as a daily cronjob. For a period of time, I also set this application to run every hour in order to spam my Miami Heat fan with Tim Duncan photos. I regret nothing.

The code for this project is relatively simple (not more than 100 actual lines of code), and setting up a Cron Job on an AWS instance can be done in less than a minute. If you have any questions about the project, feel free to e-mail me.

OneDir, Rudimentary Version of DropBox

Spring 2014 [Advanced Software Development Methods]

The problem statement here was pretty simple. Using Python, we were tasked with creating our own rudimentary version of DropBox. The problem can be broken down into two separate sections, the client side which has to recognize file changes and "ping" the server accordingly and the server side which has to take into account these changes and update the other synchronized users accordingly. For the client side, we used the Python framework WatchDog in order to monitor files. WatchDog is extremely easy to use, providing you the ability to trigger events based on the file activity that happened (for example, you can have a different event triggered if a file is created versus deleted). On the sever side, we built a Flask web server (again, very easy to use) and utilized its file transfer libraries in order to retrieve the updated files. At a basic level, we had each triggered WatchDog event hit a different endpoint of the Flask webserver so that the server could act accordingly. We also had another thread working in the background monitoring the server in case changes happen on another client's end (so that all machines that are accessing the server are updated). The Flask server was run on an AWS micro-instance.

Although I worked in a group of three other students on this project, I can confidently say that I did 95% of the coding for this project (as highlighted by this graph). Even though it wasn't the most ideal situation, especially during crunch time at the end of the semester, I really learned a lot from the project as a whole. From my experience building this OneDir system from back to front, I got comfortable picking up new libraries and frameworks that I knew nothing about and figuring out how to use them. I got really good with my debugging, google searching, and just general understanding of what to look for when figuring out new things.



TrashTalk, Android Application

Fall 2013 [MHacks]

MHacks was my second Hackathon, but the first one where I really had a grasp as to what I was doing as a programmer. I was lucky enough to fly up to Ann Arbor, Michigan with my great partner and good friend Muntaser, which helped make the experience a great one. Unfortunately we didn't have an idea going into the Hackathon, and failed to come up with something that we could both agree on for some time (we both don't consider ourselves to be the most creative). Eventually we were able to combine our love of sports with our passion for trash talking other people into a creative application that we built throughout the weekend.

TrashTalk is an application that allows the user to navigate through their contacts and select the favorite NFL teams for each contact. Then, based upon the happenings on Sunday, the application will automatically send trash talk to your friends via text message. For example, let's say that my friend Bob was a Steelers fan. If the Steelers are playing the Ravens, and the Ravens score a touchdown, a text message will be sent from my phone with some randomly generated trash talk to Bob.

Unfortunately, we were unable to integrate the sports score API before we ran out of time. The application was 90% complete, with the UI to select teams in place and the text message generation working. We have yet to go back and integrate the sports API since we have moved on to working on other projects together. At some point we will come back and tie up all the loose ends on the app.

Geography Toy for Elementary Aged Students

Fall 2011 [Introduction to Engineering Class]

My first real, overwhelming introduction to engineering and college came in the form of my first engineering class, ENGR 1620 (Introduction to Engineering). For the final project of the semester, me and a group of four other students were tasked with making a physical, educational toy for elementary aged students. As easy as this sounded, the overall scope of the project became apparent very quickly. Coordinating the programming of the toy with the physical aspect, while also making sure to fulfill the proper standards of learning in Virginia and including feedback from multiple sources, was extremely challenging. Although it there were some of bumps in the road and a lot of sleepless nights, I learned a lot from the project in the end.

On the programming end, the entire project was built in JAVA. Our toy was a Geography Toy that asked questions related to continents for students to answer. The students could physically select a continent by moving an arrow over what they believed the answer was (we coordinated with RFID tags in order to realize which continent was which). A GUI with a picture of the World Map, the question, and a students points was shown on a computer screen. As the student answered questions, the questions generated would be adjusted based on the percieved level of the student.

Unfortunately, I did not know the magic of GitHub or the importance of neat code at this point in my coding career. The code is really hard to follow and is very honed into the problem that we had. If you have any questions about the code, feel free to e-mail me.
